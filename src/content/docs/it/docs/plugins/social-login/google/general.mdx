## Differenze tra accesso online e accesso offline

Esistono diversi modi per utilizzare Google Accedi con Capacitor. Ecco una tabella che riassume le differenze tra i due:

|                         | Accesso in linea | Accesso offline |
|:---------------:|:-----:|:--------------:|
| Richiede un backend | ❌| ✅ |
| Token di accesso di lunga durata | ❌| ✅|
| Configurazione semplice | ✅| ❌|

:::note
**Modalità offline** significa che il tuo backend può accedere alle API di Google anche quando l'utente non utilizza attivamente la tua app (ad esempio, quando l'utente è "offline" dal punto di vista della tua app). I token di accesso di lunga durata abilitano questa funzionalità consentendo al back-end di chiamare le API Google per conto dell'utente in qualsiasi momento.
:::

:::caution
**La modalità offline RICHIEDE un server backend.** Quando si utilizza la modalità offline, il frontend riceve informazioni minime (principalmente solo un codice di autenticazione del server). Senza un backend per scambiare questo codice con token e informazioni sull'utente, la modalità offline non fornisce dati utili alla tua applicazione frontend.
:::

Se ancora non sai quale scegliere, considera i seguenti scenari:

1. Vuoi che l'utente effettui l'accesso immediatamente dopo avergli rilasciato un JWT personalizzato. La tua app NON chiamerà le API Google
   
   In questo caso, scegli l'accesso online.

2. La tua app chiamerà alcune API Google dal client, ma mai dal backend
   
   In questo caso, scegli l'accesso online

3. La tua app chiamerà alcune API Google dal back-end, ma solo quando l'utente utilizza attivamente l'app
   
   In questo caso, scegli l'accesso online

4. La tua app controllerà periodicamente il calendario dell'utente, anche quando non sta utilizzando attivamente l'app
   
   In questo caso, scegli l'accesso offline

## Un backend di esempio per l'accesso online

In questa parte del tutorial, mostrerò come convalidare l'utente sul tuo backend. 

Questo esempio sarà molto semplice e si baserà sulle seguenti tecnologie:

- [Dattiloscritto](https://www.typescriptlang.org/)

- [Hono](https://hono.dev/)

- [Recupero di Javascript API](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetch)

Puoi trovare il codice per questo esempio [qui](https://github.com/WcaleNieWolny/capgo-social-login-backend-demo/blob/141c01d93a85240e31a0d488a89df13c842708b1/index.ts#L135-L153)

Come puoi vedere:

<img src="/social-login-assets/vscode_auth_google.png" alt="Codice VS che mostra il codice di autenticazione Google che verifica i token" />L'idea è piuttosto semplice. Invii una semplice richiesta `GET` a `https://www.googleapis.com/oauth2/v3/tokeninfo` e questo ti restituisce se il token è valido o meno e, se lo è, ti fornisce l'e-mail dell'utente. Fornisce anche altre informazioni sul token utente
<img src="/social-login-assets/google_auth_playground_token_info.png" alt="Google OAuth Parco giochi che mostra la risposta delle informazioni sul token con i dettagli dell'utente" />

Da lì, potresti fornire all'utente il tuo JWT o emettere una sorta di cookie di sessione. Le possibilità sono infinite, per l'implementazione dell'autenticazione finale.

Se vuoi chiamare Google API, ti consiglio vivamente di guardare [Google OAuth 2.0 Playground](https://developers.google.com/oauthplayground). Da lì puoi facilmente vedere quali API puoi chiamare.

## Utilizzo dell'accesso offline con il tuo backend

Per poter utilizzare l'accesso online avrai bisogno di quanto segue:

- Un server HTTP

In questo esempio, utilizzerò le seguenti tecnologie per fornire l'accesso offline alla mia app:

- [Hono](https://hono.dev/)

- [Convalidatore Hono Zod](https://hono.dev/docs/guides/validation#with-zod)

-[Zod](https://zod.dev/)

- [Hono JWT](https://hono.dev/docs/helpers/jwt#jwt-authentication-helper)

- [LowDb](https://www.npmjs.com/package/lowdb) (un semplice database)

Il codice per questo esempio può essere trovato [qui](https://github.com/WcaleNieWolny/capgo-social-login-backend-demo/blob/aac7a8c909f650a8c2cd7f88c97f5f3c594ce9ba/index.ts#L139-L287)

Per quanto riguarda il codice cliente, è simile al seguente:

"dattiloscritto".
import { Capacitor } from '@capacitor/core';
import { GoogleLoginOfflineResponse, SocialLogin } from '@capgo/capacitor-social-login';
import { usePopoutStore } from '@/popoutStore'; // <-- specific to my app

const baseURL = "[redatto]";

funzione asincrona fullLogin() {
  attendono SocialLogin.initialize({
    Google: {
      webClientId: '[redatto]',  
      iOSClientId: '[redatto]',
      iOSServerClientId: 'Lo stesso valore di webClientId',
      modalità: 'offline' // <-- importante
    } 
  })
  const risposta = attendi SocialLogin.login({
    fornitore: "google",
    opzioni: {
      forceRefreshToken: true // <- importante
    }
  })

  if (risposta.provider === 'google') {
    const risultato = risposta.risultato come GoogleLoginOfflineResponse
    const res = attendi recupero(`${baseURL}/auth/google_offline`, {
      intestazioni: {
        "Tipo di contenuto": "application/json"
      },
      corpo: JSON.stringify({
        serverAuthCode: risultato.serverAuthCode,
        piattaforma: Capacitor.getPlatform()
      }),
      metodo: "POST"
    })

    if (res.status!== 200) {
      popoutStore.popout("Accesso completo a Google non riuscito", "controlla console");
      ritorno
    }

    const { jwt } = attendono res.json();
    const userinfo = attendi recupero(`${baseURL}/auth/get_google_user`, {
      intestazioni: {
        Autorizzazione: `Bearer ${jwt}`
      }
    })
    se (infoutente.status!== 200) {
      popoutStore.popout("Accesso completo a Google (informazioni utente) non riuscito", "controlla console");
      ritorno
    }
    popoutStore.popout("userinfo res", attendono userinfo.text());
  }
}
