L'intégration de Capgo dans votre pipeline CI/CD vous permet d'automatiser entièrement le processus de création et de déploiement des mises à jour de votre application. En tirant parti de Capgo CLI et de la version sémantique, vous pouvez garantir des déploiements cohérents et fiables et permettre une itération rapide.

## Avantages de l'intégration CI/CD

- **Automatisation** : Plus d'étapes manuelles ni de place à l'erreur humaine. L’ensemble de votre processus de création, de test et de déploiement peut être automatisé de bout en bout.

- **Cohérence** : chaque déploiement suit le même ensemble d'étapes, garantissant un processus prévisible et reproductible. Ceci est particulièrement utile lorsque plusieurs membres de l’équipe contribuent au code.

- **Itérations plus rapides** : grâce aux déploiements automatisés, vous pouvez envoyer des mises à jour plus fréquemment et en toute confiance. Plus besoin d’attendre l’assurance qualité manuelle ou les approbations de version.

## Capgo CLI

Le Capgo CLI est la clé pour intégrer Capgo dans votre flux de travail CI/CD. Il fournit des commandes pour diffuser de nouvelles versions de bundle, gérer les chaînes, etc.

La commande la plus importante pour l'intégration CI/CD est « bundle upload » :

```shell
npx @capgo/cli@latest bundle upload --channel Production --apikey YOUR_API_KEY
```

Si vous utilisez le cryptage, vous devez le fournir de l'une des manières suivantes :

**À l'aide d'un chemin de fichier de clé privée :**
```shell
npx @capgo/cli@latest bundle upload --channel Production --apikey YOUR_API_KEY --key-v2 PRIVATE_KEY_PATH
```

**Utilisation directe du contenu de la clé privée (recommandé pour CI/CD) :**
```shell
npx @capgo/cli@latest bundle upload --channel Production --apikey YOUR_API_KEY --key-data-v2 PRIVATE_KEY_CONTENT
```

**Utilisation de variables d'environnement (meilleures pratiques pour CI/CD) :**
```shell
npx @capgo/cli@latest bundle upload --channel Production --apikey YOUR_API_KEY --key-data-v2 "$CAPGO_PRIVATE_KEY"
```

### Configuration des variables d'environnement pour le chiffrement

Pour les environnements CI/CD, il est recommandé de stocker votre clé privée en tant que variable d'environnement plutôt qu'en tant que fichier. Voici comment le configurer :

1. **Obtenez le contenu de votre clé privée :**
   ```shell
   cat .capgo_key_v2 | pbcopy
   ```
   Cela copie le contenu clé dans votre presse-papiers.

2. **Ajoutez-le à votre environnement CI/CD :**
   - **GitHub Actions** : ajoutez `CAPGO_PRIVATE_KEY` aux secrets de votre référentiel
   - **GitLab CI** : Ajoutez-le en tant que variable masquée dans les paramètres de votre projet
   - **CircleCI** : Ajoutez-le comme variable d'environnement dans les paramètres de votre projet
   - **Jenkins** : ajoutez-le en tant qu'identifiant texte secret

3. **Utilisez-le dans votre pipeline :**
   ```yaml
   - run: npx @capgo/cli@latest bundle upload --channel=production --apikey=${{ secrets.CAPGO_API_KEY }} --key-data-v2 "${{ secrets.CAPGO_PRIVATE_KEY }}"
   ```

**Remarque** : L'indicateur `--key-data-v2` vous permet de transmettre le contenu de la clé privée directement sous forme de chaîne, ce qui le rend parfait pour les variables d'environnement dans les pipelines CI/CD où vous ne souhaitez pas créer de fichiers temporaires.

Cette commande télécharge la version Web actuelle sur le canal spécifié. Vous l’exécuterez généralement comme dernière étape de votre pipeline CI/CD, une fois votre création Web terminée avec succès.

## Gestion des versions avec version sémantique

La méthode recommandée pour gérer la gestion des versions avec Capgo est de définir la version dans votre fichier `capacitor.config.ts` en l'important depuis `package.json` :

```ts
import pkg from './package.json'

const config: CapacitorConfig = {
  // ... other config
  plugins: {
    CapacitorUpdater: {
      version: pkg.version,
    }
  }
}
```

Cette approche vous permet de :
1. Utilisez la version sémantique (ou tout autre outil) pour mettre à jour la version `package.json`
2. Créez votre application avec la version mise à jour automatiquement incluse
3. Téléchargez le bundle avec la bonne version

Votre flux de travail CI/CD ressemblerait à ceci :
```yaml
- run: npm ci
- run: npx semantic-release  # Updates package.json version
- run: npm run build         # Builds with new version from capacitor.config
- run: npx @capgo/cli@latest bundle upload --channel=production --apikey=${{ secrets.CAPGO_API_KEY }}
```

Voici un exemple de fichier de configuration `.releaserc` pour la version sémantique :

```json
{
  "branches": [
    "main",
    {
      "name": "beta",
      "prerelease": true
    }
  ],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    [
      "@semantic-release/git",
      {
        "assets": ["CHANGELOG.md", "package.json"],
        "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
      }
    ]
  ]
}
```

Cette configuration effectue les opérations suivantes :

1. Analyse les messages de validation pour déterminer le numéro de version suivant, conformément à la spécification Conventional Commits.
2. Génère des notes de version basées sur les validations depuis la dernière version.
3. Met à jour le fichier `CHANGELOG.md` avec les nouvelles notes de version.
4. Met à jour la version `package.json`, qui sera récupérée par votre condensateur.config.
5. Valide les fichiers `CHANGELOG.md`, `package.json` mis à jour et tout autre fichier modifié dans le référentiel.

Assurez-vous d'exécuter semantic-release avant de créer votre application afin que la version mise à jour de `package.json` soit incluse dans votre build via le fichier condensateur.config.

## Dépannage

Si vous rencontrez des problèmes avec votre intégration CI/CD Capgo, voici quelques points à vérifier :

- Clé **API** : assurez-vous que votre clé API est valide et dispose des autorisations nécessaires. Si vous utilisez une variable d’environnement, vérifiez qu’elle est correctement définie.

- **Version CLI** : assurez-vous que vous utilisez la dernière version de Capgo CLI. Les anciennes versions peuvent présenter des problèmes de compatibilité ou manquer de certaines fonctionnalités.

- **Artefacts de build** : confirmez que votre build Web génère les fichiers de sortie attendus. Le Capgo CLI a besoin d'une version Web valide pour créer un bundle.

- **Connectivité réseau** : vérifiez que votre environnement CI/CD dispose d'un accès réseau aux serveurs Capgo. Des problèmes de pare-feu ou de proxy peuvent parfois interférer avec la commande `upload`.

Si vous rencontrez toujours des problèmes, contactez l'assistance Capgo pour obtenir de l'aide. Ils peuvent vous aider à résoudre tout problème lié à votre configuration spécifique.

## Conclusion

L'intégration de Capgo dans votre pipeline CI/CD avec une gestion appropriée des versions peut considérablement rationaliser votre flux de travail de développement. En automatisant vos déploiements et votre gestion des versions via l'approche condensateur.config, vous pouvez expédier les mises à jour plus rapidement et en toute confiance.L'approche recommandée consistant à définir la version dans votre fichier `capacitor.config.ts` et à utiliser la version sémantique pour mettre à jour `package.json` fournit un processus de déploiement robuste et fiable qui vous permet de vous concentrer sur la création de fonctionnalités intéressantes plutôt que de vous soucier des étapes de publication manuelles.

Pour plus de détails sur les commandes et options Capgo CLI, consultez la [référence CLI](/docs/cli/overview). Et pour une analyse plus approfondie de la configuration de la version sémantique, consultez la [documentation de la version sémantique](https://github.com/semantic-release/semantic-release).

Bon déploiement !
