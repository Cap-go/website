## Introduzione

In questa guida vedrai come configurare il login Google con Capgo Social Login.
Per completare la configurazione ti servirà:

- Un account Google

## Configurazione generale

:::note
Questo passaggio è obbligatorio indipendentemente dalla piattaforma che decidi
di utilizzare.
:::

In questa sezione configurerai la schermata di login mostrata da Google.

<Steps>
    1. Vai su [console.cloud.google.com](https://console.cloud.google.com/)
    2. Fai clic sul selettore del progetto
        <img src="/social-login-assets/google_cons_project_selector.png" alt="Selettore del progetto nella Google Console" />

            <img src="/social-login-assets/google_cons_name_projec.png" alt="Schermata di creazione del progetto con campo nome e pulsante Create" />
        3. Assicurati di essere nel progetto corretto
            <img src="/social-login-assets/google_cons_right_proj.png" alt="Nome del progetto mostrato nel selettore dopo la selezione corretta" />
    4. Inizia la configurazione di `OAuth consent screen`
        1. Fai clic sulla barra di ricerca
            <img src="/social-login-assets/google_cons_search.png" alt="Barra di ricerca della Google Console" />
        2. Cerca `OAuth consent screen` e fai clic sul risultato
            <img src="/social-login-assets/google_cons_search_2.png" alt="Risultati di ricerca con l'opzione OAuth consent screen" />
        3. Configura la schermata di consenso
            :::note
            In questo esempio assumo che tu stia sviluppando un'app aperta al
            pubblico, quindi userò il tipo di utente `external`. Scegli il tipo

            :::

            Fai clic su `Create`
            <img src="/social-login-assets/google_cons_oauth_const_scr.png" alt="Schermata di scelta del tipo di utente per OAuth consent screen" />
    5. Compila le informazioni sulla tua app
        1. Inizia da `App Information`
            <img src="/social-login-assets/google_cons_app_inf.png" alt="Sezione App Information con nome dell'app ed email di supporto" />

            - Inserisci il tuo `App Name`
                :::caution
                **QUESTO SARÀ VISIBILE AGLI UTENTI**
                :::
            - Inserisci la `User support email`

                Puoi approfondire il significato dell'email di supporto

            <img src="/social-login-assets/google_cons_app_logo.png" alt="Sezione per il caricamento del logo dell'app" />

            <img src="/social-login-assets/google_cons_app_doma.png" alt="Sezione App domain con il campo per i domini autorizzati" />
            :::note

            :::
        4. **Devi** fornire l'email dello sviluppatore
            <img src="/social-login-assets/google_cons_dev_cont_inf.png" alt="Sezione Developer contact information con il campo email" />

            <img src="/social-login-assets/google_cons_cons_sav_cont.png" alt="Pulsante Save and continue in fondo al modulo" />

    7. Aggiungi un utente di test

            <img src="/social-login-assets/google_cons_add_test_usr.png" alt="Pulsante Add users nella sezione utenti di test" />

## Differenze tra accesso online e accesso offline

Esistono diversi modi per usare il login Google con Capacitor. La tabella
seguente riassume le differenze principali:

|                         | Accesso online | Accesso offline |
|:-----------------------:|:--------------:|:---------------:|
| Richiede un backend     | ❌             | ✅              |
| Token di lunga durata   | ❌             | ✅              |
| Configurazione semplice | ✅             | ❌              |

:::note
La **modalità offline** significa che il tuo backend può accedere alle API di
Google anche quando l'utente non sta usando attivamente la tua app. I token di
accesso a lunga durata rendono possibile questo comportamento perché permettono
al backend di chiamare le API Google per conto dell'utente in qualsiasi momento.
:::

:::caution
La **modalità offline richiede un backend**. In questo flusso il frontend
riceve pochissime informazioni, di fatto soprattutto un server auth code. Senza
un backend che scambi quel codice con token e dati utente, la modalità offline
non fornisce dati utili alla tua applicazione frontend.
:::

Se non sai ancora quale modalità scegliere, considera questi scenari:

1. Vuoi far effettuare il login all'utente e subito dopo emettere un JWT
   personalizzato. La tua app **non** chiamerà le API Google.

   In questo caso, scegli l'accesso online.

2. La tua app chiamerà alcune API Google dal client, ma mai dal backend.

   In questo caso, scegli l'accesso online.

3. La tua app chiamerà alcune API Google dal backend, ma solo mentre l'utente
   sta usando attivamente l'app.

   In questo caso, scegli l'accesso online.

4. La tua app controllerà periodicamente il calendario dell'utente anche quando
   non sta usando attivamente l'app.

   In questo caso, scegli l'accesso offline.

## Un backend di esempio per l'accesso online

In questa parte del tutorial vedrai come validare l'utente sul tuo backend.

L'esempio è volutamente semplice ed è basato sulle seguenti tecnologie:

- [TypeScript](https://www.typescriptlang.org/)
- [Hono](https://hono.dev/)
- [JavaScript Fetch API](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetch)

Puoi trovare il codice di questo esempio
[qui](https://github.com/WcaleNieWolny/capgo-social-login-backend-demo/blob/141c01d93a85240e31a0d488a89df13c842708b1/index.ts#L135-L153).

Come puoi vedere:

<img src="/social-login-assets/vscode_auth_google.png" alt="VS Code con il codice di verifica del token Google" />

L'idea è molto semplice. Invi una richiesta `GET` a
`https://www.googleapis.com/oauth2/v3/tokeninfo` e Google ti restituisce se il
token è valido oppure no. Se è valido, la risposta include anche l'email
dell'utente e altre informazioni sul token.

<img src="/social-login-assets/google_auth_playground_token_info.png" alt="Google OAuth Playground con la risposta di tokeninfo e i dati dell'utente" />

Da lì puoi emettere il tuo JWT oppure creare una sessione tramite cookie. Per
l'implementazione finale dell'autenticazione, le possibilità sono molte.

Se vuoi chiamare le API Google, ti consiglio di usare
[Google's OAuth 2.0 Playground](https://developers.google.com/oauthplayground).
Ti permette di vedere facilmente quali API puoi chiamare.

## Utilizzare l'accesso offline con il tuo backend

Per utilizzare l'accesso offline ti servirà:

- Un server HTTP

In questo esempio userò le seguenti tecnologie per fornire l'accesso offline
nella mia app:

- [Hono](https://hono.dev/)
- [Hono Zod validator](https://hono.dev/docs/guides/validation#with-zod)
- [Zod](https://zod.dev/)
- [Hono JWT](https://hono.dev/docs/helpers/jwt#jwt-authentication-helper)
- [LowDb](https://www.npmjs.com/package/lowdb) (un database semplice)

Il codice completo di questo esempio è disponibile
[qui](https://github.com/WcaleNieWolny/capgo-social-login-backend-demo/blob/aac7a8c909f650a8c2cd7f88c97f5f3c594ce9ba/index.ts#L139-L287).

Per quanto riguarda il codice client, si presenta così:

```typescript
import { Capacitor } from '@capacitor/core';
import { GoogleLoginOfflineResponse, SocialLogin } from '@capgo/capacitor-social-login';
import { usePopoutStore } from '@/popoutStore'; // <-- specific to my app

const baseURL = "[redacted]";

async function fullLogin() {
  await SocialLogin.initialize({
    google: {
      webClientId: '[redacted]',
      iOSClientId: '[redacted]',
      iOSServerClientId: 'The same value as webClientId',
      mode: 'offline' // <-- important
    }
  })
  const response = await SocialLogin.login({
    provider: 'google',
    options: {
      forceRefreshToken: true // <-- important
    }
  })

  if (response.provider === 'google') {
    const result = response.result as GoogleLoginOfflineResponse
    const res = await fetch(`${baseURL}/auth/google_offline`, {
      headers: {
        "Content-Type": "application/json"
      },
      body: JSON.stringify({
        serverAuthCode: result.serverAuthCode,
        platform: Capacitor.getPlatform()
      }),
      method: "POST"
    })

    if (res.status !== 200) {
      popoutStore.popout("Full google login failed", "check console");
      return
    }

    const { jwt } = await res.json();
    const userinfo = await fetch(`${baseURL}/auth/get_google_user`, {
      headers: {
        Authorization: `Bearer ${jwt}`
      }
    })
    if (userinfo.status !== 200) {
      popoutStore.popout("Full google (userinfo) login failed", "check console");
      return
    }
    popoutStore.popout("userinfo res", await userinfo.text());
  }
}
```
