## What can I do with API keys?

An API key can be regenerated, removed, or a new one can be added. You can also set expiration dates and create encrypted keys for enhanced security.

## How to see all API keys?

You need to go to the [API keys page](https://console.capgo.app/dashboard/apikeys/) and there you will see all your API keys.

## How to add a new API key?

To add a new API key click on the little plus button

<figure><img src="/apikeys-add.webp" alt="add apikey" /><figcaption></figcaption></figure>

and then select the permissions you want to give to the new API key. You can select:

- **Read** - the API key will be able to read all the data
- **Upload** - the API key will be able to upload new versions from the CLI and read
- **Write** - the API key will be able to modify settings, channels, devices, and upload
- **All** - the API key will be able to do everything

## How API key permissions work

API key permissions follow one of two models depending on whether your organization has the RBAC system enabled.

### Legacy mode (all organizations)

In legacy mode, an API key **inherits the permissions of the user who created it**, capped by the key's mode:

| Key mode | What the key can do                                       |
| -------- | --------------------------------------------------------- |
| `read`   | Read data only                                            |
| `upload` | Read + upload new bundles                                 |
| `write`  | Read + upload + modify channels, devices, bundle metadata |
| `all`    | Everything the owner can do in the organization           |

<Aside type="note">
  If a `super_admin` creates a `read` key, that key can only read — even though
  the owner has full rights. The mode acts as a ceiling, not a floor.
</Aside>

This means:

- If the key owner loses access to the organization, the key stops working too
- The key can never do more than the owner could do themselves
- Different modes give you a simple way to limit CI/CD keys without fine-grained control

### RBAC role bindings (organizations with RBAC enabled)

When your organization has RBAC enabled, you can assign explicit **role bindings** directly to a key (for example: `app_uploader` on a specific organization). When a key has explicit bindings:

- **Only those bindings are evaluated** — the owner's user permissions are ignored entirely
- The key is strictly limited to what its bindings allow, regardless of who owns it
- This lets you create truly minimal-privilege keys, even for super admins

If a key has no explicit bindings on an RBAC-enabled org, it falls back to legacy mode (owner's permissions capped by mode).

<Aside type="tip">
  RBAC bindings are managed from the organization settings. See [API Key
  permission system](/docs/webapp/api-key-permissions/) for a full explanation
  and examples.
</Aside>

## Encrypted API Keys

When creating a new API key, you can choose to create an **encrypted key**. Encrypted keys provide enhanced security:

- The key is hashed using SHA-256 before being stored in the database
- **The plain key is only shown once** when you create it - make sure to copy and save it immediately
- After creation, the key cannot be retrieved or viewed again
- If you lose an encrypted key, you must create a new one

This is the recommended approach for production environments as it ensures that even if the database is compromised, your actual API keys remain secure.

## API Key Expiration

You can set an **expiration date** for your API keys. This is a security best practice that limits the window of exposure if a key is compromised.

### Setting an expiration date

When creating or editing an API key, you can optionally set an expiration date. Once the expiration date passes:

- The API key will no longer work for authentication
- Any requests using the expired key will be rejected
- Expired keys are automatically cleaned up after 30 days

### Organization policies

Organizations can enforce expiration policies:

- **Require expiration**: All API keys must have an expiration date
- **Maximum expiration period**: Limit how far in the future an expiration date can be set (e.g., maximum 90 days)
- **Enforce secure API keys**: Require all new API keys to be created as encrypted/hashed keys

If your organization has these policies enabled, you will be required to set an appropriate expiration date when creating new keys.

For more information on configuring organization-wide API key policies, see [Organization Security](/docs/webapp/organization-security/).
