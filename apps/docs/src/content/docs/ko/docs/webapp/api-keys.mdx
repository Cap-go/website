## API 키로 무엇을 할 수 있나요?

API 키는 재생성하거나 삭제하거나 새로 추가할 수 있습니다. 만료일을 설정하고 보안 강화를 위한 암호화 키를 생성할 수도 있습니다.

## 모든 API 키를 어떻게 볼 수 있나요?

[API 키 페이지](https://console.capgo.app/dashboard/apikeys/)로 이동하면 모든 API 키를 볼 수 있습니다.

## 새로운 API 키를 어떻게 추가하나요?

새로운 API 키를 추가하려면 작은 플러스 버튼을 클릭하세요

<figure><img src="/apikeys-add.webp" alt="add apikey" /><figcaption></figcaption></figure>

그리고 새로운 API 키에 부여할 권한을 선택하세요. 다음과 같은 권한을 선택할 수 있습니다:

- **읽기** - API 키로 모든 데이터를 읽을 수 있습니다
- **업로드** - API 키로 CLI에서 새 버전을 업로드하고 읽을 수 있습니다
- **쓰기** - API 키로 설정, 채널, 기기를 수정하고 업로드할 수 있습니다
- **모두** - API 키로 모든 작업을 수행할 수 있습니다

## API 키 권한의 작동 방식

API 키 권한은 조직에서 RBAC 시스템이 활성화되어 있는지 여부에 따라 두 가지 모델 중 하나를 따릅니다.

### 레거시 모드 (모든 조직)

레거시 모드에서 API 키는 **생성한 사용자의 권한을 상속**하며, 키의 모드에 의해 상한이 설정됩니다:

| 키 모드 | 키로 할 수 있는 것 |
| -------- | --------------------------------------------------------- |
| `read`   | 데이터 읽기만 가능 |
| `upload` | 읽기 + 새 번들 업로드 |
| `write`  | 읽기 + 업로드 + 채널, 기기, 번들 메타데이터 수정 |
| `all`    | 조직 내에서 소유자가 할 수 있는 모든 것 |

<Aside type="note">
  `super_admin`이 `read` 키를 생성하면, 해당 키는 읽기만 가능합니다 —
  소유자가 전체 권한을 가지고 있더라도 마찬가지입니다. 모드는 하한이 아닌 상한으로 작동합니다.
</Aside>

이것은 다음을 의미합니다:

- 키 소유자가 조직에 대한 접근 권한을 잃으면, 키도 작동을 멈춥니다
- 키는 소유자 자신이 할 수 있는 것 이상의 작업을 수행할 수 없습니다
- 다양한 모드를 통해 세밀한 제어 없이도 CI/CD 키를 간단하게 제한할 수 있습니다

### RBAC 역할 바인딩 (RBAC가 활성화된 조직)

조직에서 RBAC가 활성화된 경우, 명시적인 **역할 바인딩**을 키에 직접 할당할 수 있습니다(예: 특정 조직에 대한 `app_uploader`). 키에 명시적 바인딩이 있는 경우:

- **해당 바인딩만 평가됩니다** — 소유자의 사용자 권한은 완전히 무시됩니다
- 키는 소유자가 누구인지에 관계없이 바인딩이 허용하는 범위로 엄격하게 제한됩니다
- 이를 통해 슈퍼 관리자도 최소 권한 키를 생성할 수 있습니다

RBAC 활성화 조직에서 키에 명시적 바인딩이 없는 경우, 레거시 모드(소유자 권한 + 모드 상한)로 폴백합니다.

<Aside type="tip">
  RBAC 바인딩은 조직 설정에서 관리합니다. 자세한 설명과 예시는 [API 키 권한 시스템](/docs/webapp/api-key-permissions/)을 참조하세요.
</Aside>

## 암호화 API 키

새로운 API 키를 생성할 때, **암호화 키**를 생성하도록 선택할 수 있습니다. 암호화 키는 보안을 강화합니다:

- 키는 데이터베이스에 저장되기 전에 SHA-256으로 해시됩니다
- **평문 키는 생성 시 한 번만 표시됩니다** - 즉시 복사하여 저장하세요
- 생성 후에는 키를 조회하거나 다시 볼 수 없습니다
- 암호화 키를 분실한 경우 새로 생성해야 합니다

이는 프로덕션 환경에 권장되는 방식으로, 데이터베이스가 침해되더라도 실제 API 키는 안전하게 유지됩니다.

## API 키 만료

API 키에 **만료일**을 설정할 수 있습니다. 이는 키가 유출되었을 때 노출 기간을 제한하는 보안 모범 사례입니다.

### 만료일 설정

API 키를 생성하거나 편집할 때, 선택적으로 만료일을 설정할 수 있습니다. 만료일이 지나면:

- API 키는 더 이상 인증에 사용할 수 없습니다
- 만료된 키를 사용한 모든 요청은 거부됩니다
- 만료된 키는 30일 후에 자동으로 정리됩니다

### 조직 정책

조직은 만료 정책을 강제할 수 있습니다:

- **만료 필수**: 모든 API 키에 만료일이 있어야 합니다
- **최대 만료 기간**: 만료일을 얼마나 먼 미래로 설정할 수 있는지 제한합니다(예: 최대 90일)
- **보안 API 키 강제**: 모든 새 API 키를 암호화/해시 키로 생성하도록 요구합니다

조직에서 이러한 정책이 활성화되어 있으면, 새 키를 생성할 때 적절한 만료일을 설정해야 합니다.

조직 전체 API 키 정책 설정에 대한 자세한 내용은 [조직 보안](/docs/webapp/organization-security/)을 참조하세요.
