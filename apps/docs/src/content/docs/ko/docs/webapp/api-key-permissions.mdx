Capgo는 API 키에 대해 두 가지 권한 모델을 지원합니다. 어떤 모델이 적용되는지는 조직에서 **RBAC(역할 기반 접근 제어)**가 활성화되어 있는지에 따라 달라집니다. 두 모델은 동일한 계정 내 서로 다른 조직에서 공존할 수 있습니다.

## 개요

| 모델 | 적용되는 경우 | 권한 결정 방법 |
| -------------------- | -------------------------------------------- | ------------------------------------------------------ |
| **레거시 모드** | 모든 조직 (기본값) | 소유자의 권한, 키 모드에 의한 상한 |
| **RBAC 역할 바인딩** | RBAC가 활성화된 조직, 키에 바인딩 있음 | 키 자체의 역할 바인딩만 (소유자 무시) |

<Aside type="note">
  RBAC가 활성화된 조직에서 명시적 역할 바인딩이 **없는** 키는
  해당 조직에서 레거시 모드로 폴백합니다.
</Aside>

## 레거시 모드

레거시 모드는 기본값이며 모든 조직에서 작동합니다. 권한 로직은 간단합니다:

1. 키에는 **모드**가 있습니다 (`read`, `upload`, `write`, 또는 `all`)
2. 키는 **사용자**(생성자/소유자)에 속합니다
3. 해당 사용자는 조직 내에서 **역할**을 가집니다 (`read`, `upload`, `write`, `admin`, `super_admin`)
4. 키의 유효 권한은 모드와 사용자 역할의 **교집합**입니다

### 4가지 키 모드

키 모드는 단계적 스키마를 따르며, 각 레벨은 하위 레벨의 모든 것을 포함합니다:

<figure><img style="margin-left: auto; margin-right: auto" src="/capgo_apikeys_diagram.webp" alt="API key modes diagram showing the graduated permission levels" /><figcaption></figcaption></figure>

| 모드 | 기능 |
| -------- | ----------------------------------------------------------------------------- |
| `read`   | 앱, 번들, 채널, 기기, 통계, 로그 읽기 |
| `upload` | `read`의 모든 것 + 새 번들 업로드 |
| `write`  | `upload`의 모든 것 + 채널, 기기, 번들 메타데이터 수정 |
| `all`    | `write`의 모든 것 + 앱 삭제, 조직 설정 및 멤버 관리 |

### 실제 작동 방식

조직에서 `admin`인 사용자를 생각해 보겠습니다:

| 키 모드 | 유효 권한 |
| ---------- | ------------------------------------------------------------------------- |
| `read`     | 읽기만 가능 (모드가 역할보다 제한적) |
| `upload`   | 읽기 + 업로드 (모드가 역할보다 제한적) |
| `write`    | 읽기 + 업로드 + 수정 (역할의 레벨과 일치) |
| `all`      | 실질적으로 `write`와 동일 (admin은 앱 삭제나 결제 관리 불가) |

<Aside type="caution">
  키 소유자가 **조직에서 제거**되거나 역할이 다운그레이드되면,
  키의 유효 권한이 즉시 변경됩니다. 키는 소유자가 할 수 있는 것을 초과할 수 없습니다.
</Aside>

### 일반적인 레거시 사용 사례

- **CI/CD 파이프라인**: `upload` 키를 생성하여 CI가 번들을 배포할 수 있지만 채널 수정이나 앱 삭제는 불가능하게 합니다
- **모니터링 통합**: 통계만 가져오면 되는 대시보드용으로 `read` 키를 사용합니다
- **배포 자동화**: 번들을 업로드하고 채널 버전을 전환하는 스크립트용으로 `write` 키를 사용합니다

## RBAC 역할 바인딩

조직에서 RBAC가 활성화된 경우, **역할 바인딩**을 API 키에 직접 할당할 수 있습니다. 이는 단순한 모드 기반 시스템을 넘어서는 세밀한 제어를 제공합니다.

### 역할 바인딩이란?

역할 바인딩은 특정 **역할**(예: `app_reader`, `app_uploader`, `org_admin`)을 **스코프**(조직, 앱, 또는 채널) 내의 **주체**(사용자, 그룹, 또는 API 키)에 할당합니다.

API 키의 경우, "이 키는 조직 X에 대해 `app_uploader` 역할을 가진다"라고 지정할 수 있습니다 — 키의 소유자가 무엇을 할 수 있는지와는 독립적으로.

### 바인딩 우선순위

이것은 이해해야 할 가장 중요한 개념입니다:

**API 키에 명시적 역할 바인딩이 있으면, 해당 바인딩만 평가됩니다. 소유자의 권한은 완전히 무시됩니다.**

이는 의도적인 것이며 중요한 보안 보장을 제공합니다: 제한된 키는 소유자가 `super_admin`이더라도 제한된 상태를 유지합니다.

```
키에 역할 바인딩이 있는가? ──예──> 키 바인딩만 평가 ──> 허용/거부
                          │
                         아니오
                          │
                          v
                  레거시 모드로 폴백
                  (소유자 권한 + 키 모드)
```

### 시나리오 예시

**시나리오 1: 슈퍼 관리자의 제한된 CI 키**

`super_admin`이 프로덕션 조직에 대해 명시적 `app_uploader` 바인딩을 가진 키를 생성한 경우:

- 키는 해당 조직에 번들을 업로드할 수 있습니다
- 키는 앱 삭제, 설정 변경, 멤버 관리를 **할 수 없습니다**
- 슈퍼 관리자의 광범위한 권한은 이 키에 무관합니다

**시나리오 2: 읽기 전용 감사 키**

`super_admin`이 `app_reader` 바인딩을 가진 키를 생성한 경우:

- 키는 앱 데이터, 번들, 채널만 읽을 수 있습니다
- 키는 업로드, 수정, 삭제가 불가능합니다
- 모니터링 도구나 외부 대시보드에 적합합니다

**시나리오 3: RBAC 조직에서 바인딩이 없는 키**

키에 RBAC 식별자는 있지만 명시적 역할 바인딩이 없는 경우, 레거시 동작으로 폴백합니다:

- 조직 내 소유자의 RBAC 역할이 평가됩니다
- 키 모드(`read`/`upload`/`write`/`all`)가 상한으로 작동합니다

### 역할 바인딩 관리

{/* TODO: screenshot of RBAC bindings UI - apikeys-rbac-bindings.webp */}

API 키의 역할 바인딩은 조직 설정에서 관리합니다. 다음을 수행할 수 있습니다:

1. 키가 가진 바인딩 확인
2. 새로운 역할 바인딩 추가 (예: 특정 조직에 대한 `app_uploader`)
3. 기존 바인딩을 제거하여 접근을 제한하거나 취소

<Aside type="tip">
  RBAC가 활성화된 상태에서 콘솔 UI를 통해 키를 생성하면, 선택에 따라 바인딩이
  자동으로 설정됩니다. REST API를 통해 생성된 키는 바인딩을 별도로 구성할 수 있습니다.
</Aside>

## 스코프 제한

모드와 역할 바인딩 외에도, API 키는 추가 필터로 작동하는 두 가지 **스코프 제한**을 지원합니다:

### `limited_to_orgs`

키를 특정 조직 내에서만 작동하도록 제한합니다. 키가 이 목록에 없는 조직에 접근하려고 하면, 모드나 바인딩이 허용하더라도 요청이 거부됩니다.

### `limited_to_apps`

키를 특정 앱(앱 ID)에서만 작동하도록 제한합니다. 단일 앱에만 배포할 수 있는 키를 생성하는 데 유용합니다.

<Aside type="note">
  스코프 제한은 **추가 필터**이며, 권한 부여가 아닙니다. 키는 작업을
  수행하기 위해 적절한 모드/바인딩이 여전히 필요합니다 — 스코프는
  해당 권한이 적용되는 범위만 제한합니다.
</Aside>

## 보안 비교

| 측면 | 레거시 모드 | RBAC 역할 바인딩 |
| ----------------------------- | -------------------------------------- | ------------------------------------------ |
| 권한 출처 | 키 소유자의 조직 역할 + 키 모드 | 키 자체의 역할 바인딩만 |
| 소유자가 조직을 떠남 | 키가 작동 중지 | 키가 계속 작동 (바인딩은 독립적) |
| 소유자 역할 다운그레이드 | 키 권한이 즉시 축소 | 키 권한에 영향 없음 |
| 최소 권한 실현 가능성 | 모드 세분성에 제한 (4단계) | 권한 단위의 세밀한 제어 |
| 설정 복잡도 | 간단 (모드만 선택) | RBAC 구성 필요 |

<Aside type="caution">
  레거시 모드에서 사용자가 조직에서 제거되면, 해당 사용자의 모든 API 키는
  즉시 해당 조직에 대한 접근 권한을 잃습니다. RBAC 바인딩에서는 키 자체의
  바인딩이 독립적입니다 — 그러나 사용자 불일치 검사에 의해 키 소유자가
  인증된 사용자여야 합니다.
</Aside>

## CLI 동작

Capgo CLI는 두 가지 권한 모델 모두에서 자동으로 작동합니다:

- `app add`, `init`, `organization set`, `organization members`와 같은 명령은 레거시와 RBAC 경로를 모두 준수하는 `cli_check_permission` 함수를 사용합니다
- `bundle upload`, `bundle list`, `channel` 작업과 같은 명령은 레거시 권한 경로를 사용합니다
- CLI는 조직의 구성에 따라 어떤 시스템이 적용되는지 자동으로 감지합니다 — 수동 전환이 필요 없습니다

CLI 명령의 전체 목록과 필요한 권한은 [CLI 명령 레퍼런스](/docs/cli/reference/login/)를 참조하세요.
