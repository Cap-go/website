Capgo 支持两种 API 密钥权限模型。适用哪种模型取决于您的组织是否启用了 **RBAC（基于角色的访问控制）**。两种模型可以在同一账户的不同组织中共存。

## 概述

| 模型 | 适用场景 | 权限确定方式 |
| -------------------- | -------------------------------------------- | ------------------------------------------------------ |
| **传统模式** | 所有组织（默认） | 所有者的权限，受密钥模式上限限制 |
| **RBAC 角色绑定** | 启用了 RBAC 的组织，密钥有绑定 | 仅密钥自身的角色绑定（忽略所有者） |

<Aside type="note">
  如果启用了 RBAC 的组织中，密钥**没有**明确的角色绑定，
  该密钥将在该组织中回退到传统模式。
</Aside>

## 传统模式

传统模式是默认模式，适用于所有组织。权限逻辑很简单：

1. 密钥有一个**模式**（`read`、`upload`、`write` 或 `all`）
2. 密钥属于一个**用户**（其创建者/所有者）
3. 该用户在组织中有一个**角色**（`read`、`upload`、`write`、`admin`、`super_admin`）
4. 密钥的有效权限是模式和用户角色的**交集**

### 四种密钥模式

密钥模式遵循分级架构，每个级别包含其下所有级别的功能：

<figure><img style="margin-left: auto; margin-right: auto" src="/capgo_apikeys_diagram.webp" alt="API key modes diagram showing the graduated permission levels" /><figcaption></figcaption></figure>

| 模式 | 功能 |
| -------- | ----------------------------------------------------------------------------- |
| `read`   | 读取应用、包、频道、设备、统计、日志 |
| `upload` | `read` 的所有功能 + 上传新包 |
| `write`  | `upload` 的所有功能 + 修改频道、设备、包元数据 |
| `all`    | `write` 的所有功能 + 删除应用、管理组织设置和成员 |

### 实际工作方式

考虑一个在组织中担任 `admin` 的用户：

| 密钥模式 | 有效权限 |
| ---------- | ------------------------------------------------------------------------- |
| `read`     | 仅读取（模式比角色更具限制性） |
| `upload`   | 读取 + 上传（模式比角色更具限制性） |
| `write`    | 读取 + 上传 + 修改（与角色级别匹配） |
| `all`      | 实际上与 `write` 相同（admin 无法删除应用或管理计费） |

<Aside type="caution">
  如果密钥所有者被**从组织中移除**或其角色被降级，
  密钥的有效权限会立即更改。密钥永远不能超越其所有者的能力。
</Aside>

### 常见传统模式用例

- **CI/CD 管道**: 创建 `upload` 密钥，使 CI 可以部署包但无法修改频道或删除应用
- **监控集成**: 为仅需获取统计信息的仪表板使用 `read` 密钥
- **部署自动化**: 为上传包和切换频道版本的脚本使用 `write` 密钥

## RBAC 角色绑定

当您的组织启用了 RBAC 时，您可以将**角色绑定**直接分配给 API 密钥。这提供了超越简单模式系统的细粒度控制。

### 什么是角色绑定？

角色绑定将特定**角色**（如 `app_reader`、`app_uploader` 或 `org_admin`）分配给**范围**（组织、应用或频道）内的**主体**（用户、组或 API 密钥）。

对于 API 密钥，这意味着您可以指定："此密钥在组织 X 上具有 `app_uploader` 角色" — 独立于密钥所有者的能力。

### 绑定优先级

这是需要理解的最重要概念：

**当 API 密钥具有明确的角色绑定时，仅评估这些绑定。所有者的权限被完全忽略。**

这是有意为之的，提供了关键的安全保证：受限密钥保持受限状态，即使其所有者是 `super_admin`。

```
密钥有角色绑定？ ──是──> 仅评估密钥绑定 ──> 允许/拒绝
                  │
                 否
                  │
                  v
          回退到传统模式
          （所有者权限 + 密钥模式）
```

### 场景示例

**场景 1: 超级管理员的受限 CI 密钥**

`super_admin` 创建了一个在生产组织上具有明确 `app_uploader` 绑定的密钥：

- 密钥可以向该组织上传包
- 密钥**无法**删除应用、更改设置或管理成员
- 超级管理员的广泛权限与此密钥无关

**场景 2: 只读审计密钥**

`super_admin` 创建了一个具有 `app_reader` 绑定的密钥：

- 密钥只能读取应用数据、包、频道
- 密钥无法上传、修改或删除任何内容
- 非常适合监控工具或外部仪表板

**场景 3: RBAC 组织中无绑定的密钥**

如果密钥有 RBAC 标识符但没有明确的角色绑定，则回退到传统行为：

- 评估所有者在组织中的 RBAC 角色
- 密钥模式（`read`/`upload`/`write`/`all`）充当上限

### 管理角色绑定

{/* TODO: screenshot of RBAC bindings UI - apikeys-rbac-bindings.webp */}

API 密钥的角色绑定从组织设置中管理。您可以：

1. 查看密钥具有的绑定
2. 添加新的角色绑定（例如，特定组织上的 `app_uploader`）
3. 删除现有绑定以限制或撤销访问

<Aside type="tip">
  在启用 RBAC 的情况下通过控制台 UI 创建密钥时，绑定会根据您的选择
  自动设置。通过 REST API 创建的密钥可以单独配置绑定。
</Aside>

## 范围限制

除了模式和角色绑定外，API 密钥还支持两种作为附加过滤器的**范围限制**：

### `limited_to_orgs`

将密钥限制为仅在特定组织内工作。如果密钥用于访问不在此列表中的组织，即使模式或绑定允许，请求也会被拒绝。

### `limited_to_apps`

将密钥限制为仅与特定应用（按应用 ID）一起工作。适用于创建只能部署到单个应用的密钥。

<Aside type="note">
  范围限制是**附加过滤器**，而非权限授予。密钥仍然需要适当的模式/绑定
  来执行操作 — 范围只是限制这些权限适用的范围。
</Aside>

## 安全对比

| 方面 | 传统模式 | RBAC 角色绑定 |
| ----------------------------- | -------------------------------------- | ------------------------------------------ |
| 权限来源 | 密钥所有者的组织角色 + 密钥模式 | 仅密钥自身的角色绑定 |
| 所有者离开组织 | 密钥停止工作 | 密钥继续工作（绑定是独立的） |
| 所有者角色降级 | 密钥权限立即缩减 | 对密钥权限无影响 |
| 最小权限可实现性 | 受模式粒度限制（4个级别） | 按权限的细粒度控制 |
| 设置复杂度 | 简单（只需选择模式） | 需要 RBAC 配置 |

<Aside type="caution">
  在传统模式下，如果用户被从组织中移除，其所有 API 密钥将立即失去
  对该组织的访问权限。使用 RBAC 绑定时，密钥自身的绑定是独立的 ——
  但用户不匹配检查仍要求密钥所有者是经过身份验证的用户。
</Aside>

## CLI 行为

Capgo CLI 自动适用于两种权限模型：

- `app add`、`init`、`organization set` 和 `organization members` 等命令使用同时遵循传统和 RBAC 路径的 `cli_check_permission` 函数
- `bundle upload`、`bundle list` 和 `channel` 操作等命令使用传统权限路径
- CLI 根据组织的配置自动检测适用哪种系统 — 无需手动切换

有关 CLI 命令的完整列表及其所需权限，请参阅 [CLI 命令参考](/docs/cli/reference/login/)。
