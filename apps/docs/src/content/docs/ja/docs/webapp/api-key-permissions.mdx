Capgoは、APIキーに対して2つの権限モデルをサポートしています。どのモデルが適用されるかは、組織で**RBAC（ロールベースアクセス制御）**が有効かどうかによります。両方のモデルは、同じアカウント内の異なる組織間で共存できます。

## 概要

| モデル | 適用される場合 | 権限の決定方法 |
| -------------------- | -------------------------------------------- | ------------------------------------------------------ |
| **レガシーモード** | すべての組織（デフォルト） | オーナーの権限、キーモードによる上限付き |
| **RBACロールバインディング** | RBACが有効な組織、キーにバインディングあり | キー自身のロールバインディングのみ（オーナーは無視） |

<Aside type="note">
  RBAC有効な組織で、明示的なロールバインディングが**ない**キーの場合、
  その組織ではレガシーモードにフォールバックします。
</Aside>

## レガシーモード

レガシーモードはデフォルトで、すべての組織で動作します。権限のロジックはシンプルです：

1. キーには**モード**があります（`read`、`upload`、`write`、または`all`）
2. キーは**ユーザー**（作成者/オーナー）に属します
3. そのユーザーは組織内で**ロール**を持っています（`read`、`upload`、`write`、`admin`、`super_admin`）
4. キーの有効な権限は、モードとユーザーのロールの**交差部分**です

### 4つのキーモード

キーモードは段階的なスキーマに従い、各レベルはそれ以下のすべてを含みます：

<figure><img style="margin-left: auto; margin-right: auto" src="/capgo_apikeys_diagram.webp" alt="API key modes diagram showing the graduated permission levels" /><figcaption></figcaption></figure>

| モード | 機能 |
| -------- | ----------------------------------------------------------------------------- |
| `read`   | アプリ、バンドル、チャネル、デバイス、統計、ログの読み取り |
| `upload` | `read`のすべて + 新しいバンドルのアップロード |
| `write`  | `upload`のすべて + チャネル、デバイス、バンドルメタデータの変更 |
| `all`    | `write`のすべて + アプリの削除、組織設定とメンバーの管理 |

### 実際の動作

組織で`admin`であるユーザーを考えてみましょう：

| キーモード | 有効な権限 |
| ---------- | ------------------------------------------------------------------------- |
| `read`     | 読み取りのみ（モードがロールより制限的） |
| `upload`   | 読み取り + アップロード（モードがロールより制限的） |
| `write`    | 読み取り + アップロード + 変更（ロールのレベルと一致） |
| `all`      | 実質的には`write`と同じ（adminはアプリの削除や課金管理ができない） |

<Aside type="caution">
  キーのオーナーが**組織から削除**されたり、ロールがダウングレードされた場合、
  キーの有効な権限は即座に変更されます。キーはオーナーができること以上のことはできません。
</Aside>

### レガシーモードの一般的なユースケース

- **CI/CDパイプライン**: `upload`キーを作成して、CIがバンドルをデプロイできるが、チャネルの変更やアプリの削除はできないようにする
- **監視統合**: 統計の取得のみが必要なダッシュボード用に`read`キーを使用する
- **デプロイ自動化**: バンドルのアップロードとチャネルバージョンの切り替えを行うスクリプト用に`write`キーを使用する

## RBACロールバインディング

組織でRBACが有効な場合、**ロールバインディング**をAPIキーに直接割り当てることができます。これにより、シンプルなモードベースのシステムを超えたきめ細かい制御が可能になります。

### ロールバインディングとは？

ロールバインディングは、特定の**ロール**（`app_reader`、`app_uploader`、`org_admin`など）を、**スコープ**（組織、アプリ、またはチャネル）内の**プリンシパル**（ユーザー、グループ、またはAPIキー）に割り当てます。

APIキーの場合、これは「このキーは組織Xに対して`app_uploader`ロールを持つ」と指定できることを意味します — キーのオーナーができることとは独立して。

### バインディングの優先順位

これは理解すべき最も重要な概念です：

**APIキーに明示的なロールバインディングがある場合、そのバインディングのみが評価されます。オーナーの権限は完全に無視されます。**

これは意図的なもので、重要なセキュリティ保証を提供します：制限されたキーは、オーナーが`super_admin`であっても制限されたままです。

```
キーにロールバインディングがある？ ──はい──> キーバインディングのみを評価 ──> 許可/拒否
                                │
                               いいえ
                                │
                                v
                       レガシーモードにフォールバック
                       （オーナー権限 + キーモード）
```

### シナリオ例

**シナリオ1: スーパー管理者の制限付きCIキー**

`super_admin`が本番組織に対して明示的な`app_uploader`バインディングを持つキーを作成した場合：

- キーはその組織にバンドルをアップロードできます
- キーはアプリの削除、設定の変更、メンバーの管理は**できません**
- スーパー管理者の広範な権限はこのキーには無関係です

**シナリオ2: 読み取り専用の監査キー**

`super_admin`が`app_reader`バインディングを持つキーを作成した場合：

- キーはアプリデータ、バンドル、チャネルの読み取りのみが可能です
- キーはアップロード、変更、削除はできません
- 監視ツールや外部ダッシュボードに最適です

**シナリオ3: RBAC組織でバインディングのないキー**

キーにRBAC識別子はあるが明示的なロールバインディングがない場合、レガシー動作にフォールバックします：

- 組織内のオーナーのRBACロールが評価されます
- キーモード（`read`/`upload`/`write`/`all`）が上限として機能します

### ロールバインディングの管理

{/* TODO: screenshot of RBAC bindings UI - apikeys-rbac-bindings.webp */}

APIキーのロールバインディングは、組織の設定から管理します。以下のことが可能です：

1. キーが持つバインディングを表示する
2. 新しいロールバインディングを追加する（例：特定の組織に対する`app_uploader`）
3. 既存のバインディングを削除してアクセスを制限または取り消す

<Aside type="tip">
  RBAC有効時にコンソールUIからキーを作成すると、選択に基づいてバインディングが
  自動的に設定されます。REST API経由で作成されたキーは、バインディングを個別に設定できます。
</Aside>

## スコープ制限

モードとロールバインディングに加えて、APIキーは追加のフィルターとして機能する2つの**スコープ制限**をサポートしています：

### `limited_to_orgs`

キーを特定の組織内でのみ動作するように制限します。キーがこのリストにない組織にアクセスしようとすると、モードやバインディングで許可されていてもリクエストは拒否されます。

### `limited_to_apps`

キーを特定のアプリ（アプリID）でのみ動作するように制限します。単一のアプリにのみデプロイできるキーを作成するのに便利です。

<Aside type="note">
  スコープ制限は**追加フィルター**であり、権限の付与ではありません。キーは
  アクションを実行するために適切なモード/バインディングが必要です — スコープは
  それらの権限が適用される場所を制限するだけです。
</Aside>

## セキュリティ比較

| 観点 | レガシーモード | RBACロールバインディング |
| ----------------------------- | -------------------------------------- | ------------------------------------------ |
| 権限のソース | キーオーナーの組織ロール + キーモード | キー自身のロールバインディングのみ |
| オーナーが組織を離脱 | キーが動作しなくなる | キーは動作し続ける（バインディングは独立） |
| オーナーのロールがダウングレード | キーの権限が即座に縮小 | キーの権限に影響なし |
| 最小権限の実現可能性 | モードの粒度に制限（4レベル） | パーミッション単位のきめ細かい制御 |
| セットアップの複雑さ | シンプル（モードを選ぶだけ） | RBAC設定が必要 |

<Aside type="caution">
  レガシーモードでは、ユーザーが組織から削除されると、そのユーザーのすべてのAPIキーは
  即座にその組織へのアクセスを失います。RBACバインディングでは、キー自身のバインディングは
  独立しています — ただし、ユーザー不一致チェックにより、キーのオーナーが
  認証済みユーザーである必要があります。
</Aside>

## CLIの動作

Capgo CLIは両方の権限モデルで自動的に動作します：

- `app add`、`init`、`organization set`、`organization members`などのコマンドは、レガシーとRBACの両方のパスを尊重する`cli_check_permission`関数を使用します
- `bundle upload`、`bundle list`、`channel`操作などのコマンドは、レガシー権限パスを使用します
- CLIは組織の設定に基づいてどのシステムが適用されるかを検出します — 手動での切り替えは不要です

CLIコマンドの完全なリストと必要な権限については、[CLIコマンドリファレンス](/docs/cli/reference/login/)を参照してください。
